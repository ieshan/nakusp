## Essential Commands

**Always use `./dev.sh`. Do not run `go` commands directly** for cross-module operations.

```bash
# Setup (generates go.work, syncs deps, tidies all modules)
./dev.sh setup

# Run all tests locally (root + redis + sqlite modules)
./dev.sh test

# Run tests in Docker (includes Redis server for integration tests)
./dev.sh test-docker

# Full CI simulation (vet -> build -> test)
./dev.sh ci

# Run go mod tidy in all modules + workspace sync
./dev.sh tidy
```

### Running specific tests

```bash
# Specific package in the root module
(cd . && go test -v ./transports)

# Specific test case in root
(cd . && go test -v ./transports -run TestFakeTransport)

# Redis transport tests (requires REDIS_URI or Docker)
(cd ./transports/redis && go test -v ./...)

# SQLite transport tests
(cd ./transports/sqlite && go test -v ./...)
```

## Build/Test Requirements

- Root module (`github.com/ieshan/nakusp`) is pure Go, zero external transport deps
- Redis submodule (`transports/redis`) requires `github.com/redis/go-redis/v9`
- SQLite submodule (`transports/sqlite`) requires `github.com/mattn/go-sqlite3` (CGO)
- Redis integration tests require a running Redis server; set `REDIS_URI` or use `test-docker`
- `go.work` must exist for `./...` to discover submodules; `./dev.sh setup` creates it

## Code Patterns and Conventions

### Transport interface

All transports implement `models.Transport`. Key methods block until context cancelled:
- `Heartbeat` and `Consume` are long-running goroutines
- `ConsumeAll` fetches all jobs, closes channel, then returns
- `FakeTransport` mirrors this blocking behavior for consistent testing

### Error style

- Wrap errors with context: `fmt.Errorf("transport: op: %w", err)`
- Transport layers return explicit errors for malformed data

### Redis transport specifics

- Uses `transports.GetPayload(job)` for string serialization (colon-separated)
- Lua script (`fetchTasks`) handles atomic job claiming with distributed lock
- `HSetNX` + `HExpire` for worker heartbeat with auto-expiration

### SQLite transport specifics

- Transaction-based job locking (`fetchOnce`) prevents duplicate processing
- `db.SetMaxOpenConns(1)` to avoid writer contention
- `ConsumeAll` queries `status = 'queued'` and streams results via channel

## Testing Approach

- Tests are colocated `*_test.go` in each module
- Root `transports/transport_test.go` tests only FakeTransport
- Redis and SQLite have dedicated test files in their submodules
- Redis tests are gated by `REDIS_URI` env var; skipped if absent
- SQLite-specific regression tests:
  - `sqlite_atomic_claim_single_delivery` — verifies only one consumer claims a job
  - `sqlite_dlq_not_requeued` — verifies DLQ jobs are not re-fetched

## Gotchas and Non-Obvious Behaviors

- `go.work` is gitignored. If `./...` or `go test ./...` silently skips submodule tests, run `./dev.sh setup` first.
- The root `transports` package comment currently mentions Redis/SQLite — update it after modularization.
- `NewNakusp(nil, nil)` defaults to FakeTransport. If a user passes transports, they must include one explicitly.
- SQLite transport uses `database/sql` + `mattn/go-sqlite3` (CGO required). SQLite submodule tests will fail if CGO is disabled.
- Redis transport's `parseJobPayload` is package-private and used only internally; `GetPayload` is the exported counterpart.
- Docker Compose mounts `./:/app` and generates its own `go.work` inside the container. Running compose does not require host `./dev.sh setup`.

## Practical Agent Workflow

1. Prefer `./dev.sh test` for root-only changes.
2. If touching `transports/redis`, run `./dev.sh test-docker` or `(cd transports/redis && go test -v ./...)` with `REDIS_URI` set.
3. If touching `transports/sqlite`, run `(cd transports/sqlite && go test -v ./...)`.
4. If adding a new transport submodule:
   - Create `transports/<name>/go.mod` with `replace github.com/ieshan/nakusp => ../../`
   - Update `dev.sh` MODULES array
   - Update `compose.yml` command to include the new module in `go work init`
   - Run `./dev.sh setup` to regenerate `go.work`
5. Keep interface assertions and error wrapping style consistent across all transports.
6. Never add external dependencies to the root `go.mod` — they belong in the relevant transport submodule.
